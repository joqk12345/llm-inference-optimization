# 第9章：投机采样

> **💰 成本影响**（常见量级，强依赖草稿模型质量/接受率/实现与工作负载）
> - **加速潜力**：在合适条件下，投机采样能显著提升 decode 吞吐
> - **成本结构**：加速来自“少跑大模型的步数”，但会引入草稿模型开销与验证开销
> - **适用边界**：长文本与稳定分布更容易受益；短回答或强约束格式有时收益有限

## 简介

在前面的章节中，我们学习了 GPU、KV、调度、量化等核心杠杆。这些技术大多围绕“把每一步做得更快/更省”。

但自回归生成有一个根本限制:**只能逐个生成 token**。即使 GPU 很快,生成 100 个 tokens 也需要 100 次 forward pass。对于长文本生成 (文章、代码、报告),这成为了一个明显的瓶颈。

**投机采样 (Speculative Sampling)** 是打破这个限制的关键技术。它的核心思想是:**用一个小模型快速"猜测"后面的 tokens,然后用大模型"验证"这些猜测**。如果猜测正确,就节省了计算;如果猜测错误,就丢弃结果重新计算。

这种“投机-验证”机制在合适条件下可以带来显著加速，但它不是白嫖：收益取决于草稿模型的接受率、验证策略以及实现细节。本章将深入讲解：
- 为什么自回归生成慢,投机采样如何解决
- 投机采样的原理和流程
- 草稿模型的选择策略
- 以 Eagle 类草稿模型为例的端到端落地思路（部署、压测、调参）
- 自训练草稿模型的端到端流程（数据、训练、验证、上线）

**学完本章，你将能判断投机采样是否适合你的负载，并能用指标把收益与成本算清楚。**

## 9.1 生成加速的基本思路

**背景**：大多数推理优化（KV、调度、量化）是在“把每一步做得更省”。投机采样则试图回答另一个问题：能不能少做一些“大模型的步骤”？这也是它收益可能很大、但边界也更清晰的原因。

**决策**：在深入原理前，先记住两个判断：

- 投机采样的收益来自接受率（draft 猜对的比例），而不是来自草稿模型“跑得多快”本身。
- 你的负载越像“长 decode、分布稳定、可接受额外复杂度”，越可能受益；反之收益可能有限甚至负收益。

### 9.1.1 为什么自回归生成慢

**自回归生成的限制**:
```
每个 token 的生成都依赖于之前的 tokens:

Token 0:
  输入: [t0]
  输出: [t0, t1]

Token 1:
  输入: [t0, t1]
  输出: [t0, t1, t2]

Token 2:
  输入: [t0, t1, t2]
  输出: [t0, t1, t2, t3]

...

Token n:
  输入: [t0, t1, ..., tn]
  输出: [t0, t1, ..., tn, t(n+1)]

问题:
  - 每个 token 需要 1 次 forward pass
  - 无法并行 (因为依赖关系)
  - 生成 100 tokens = 100 次 forward pass
```

**性能瓶颈**:
```
单个 forward pass 的时间:
  - 取决于模型规模、上下文长度、并发与硬件
  - 对许多在线负载来说，单步 decode 往往是毫秒级到几十毫秒级
  - 生成 N 个 tokens 通常需要 N 次 decode step，因此总耗时随输出长度线性增长

说明：以上用于解释“为什么慢”的结构性原因。真实数值应以你的模型、框架与负载压测为准（TTFT/TPOT/P95/P99）。
```

---

### 9.1.2 并行化生成的挑战

**为什么不能直接并行?**
```
错误的并行尝试:
  同时生成 token 1-10
  → 输出: [t0, t1', t2', ..., t10']

问题:
  - token 1' 依赖于 t0,但没有 t0 的上下文
  - token 2' 依赖于 t0, t1',但 t1' 可能错误
  - 错误会累积
  - 最终输出不连贯
```

**因果约束**:
```
Token i 只能依赖于 tokens 0, 1, ..., i-1
Token i 不能依赖于 tokens i+1, i+2, ...

这是 LLM 的基本设计
有助于保证输出的连贯性（但也带来“必须逐 token 生成”的串行约束）
```

---

### 9.1.3 投机执行的概念

**来自 CPU 的灵感**:
```
CPU 分支预测:
  CPU 遇到条件分支 (if-else)
  → 先"猜测"会走哪个分支
  → 提前执行猜测的分支
  → 验证猜测是否正确
  → 如果正确,节省时间
  → 如果错误,丢弃结果重新执行

关键洞察:
  - 猜测命中率通常较高,但依模型与任务而定
  - 即使有一定错误率,整体仍可能加速
```

**应用到 LLM 生成**:
```
传统方式:
  Token 0 → Token 1 → Token 2 → ... → Token 100
  (100 次 forward pass)

投机采样:
  Token 0 → [猜测 Token 1-10]
  → 验证 Token 1-10
  → 如果正确,节省了 9 次 forward pass
  → 如果错误,重新计算
```

## 9.2 投机采样原理

**背景**：投机采样可以被理解为“用便宜模型预取，用贵模型验收”。它把 decode 的串行步数从“必须做 N 次大模型 forward”变成“做更少的大模型 forward + 一些小模型 forward + 验证开销”。

**决策**：它是否划算，主要取决于三件事：

- 草稿模型的接受率（acceptance rate）是否足够高
- 验证策略是否高效（验证开销是否会吞掉收益）
- 你的负载是否真的 decode-heavy（如果主要时间在 prefill，收益会变小）

### 9.2.1 核心思想: 小模型先行

**两模型架构**:
```
主模型 (Main Model):
  - 大模型 (如 Llama-2-70B)
  - 准确但慢
  - 负责验证草稿

草稿模型 (Draft Model):
  - 小模型 (如 Llama-2-7B)
  - 快但不够准确
  - 负责生成草稿
```

**工作流程**:
```
步骤 1: 草稿模型快速生成
  输入: [t0]
  草稿模型: 生成 [t1, t2, ..., t10] (快速)
  时间: 1 次 forward pass (小模型)

步骤 2: 主模型并行验证
  输入: [t0, t1, t2, ..., t10]
  主模型: 并行验证所有 tokens (1 次 forward pass)
  输出: [t0, t1', t2', ..., t10']

步骤 3: 比对结果
  如果 t1' == t1, t2' == t2, ..., t10' == t10:
    → 全部正确! ✅
    → 接受草稿,节省时间
  否则:
    → 找到第一个不匹配的 token (如 t3)
    → 丢弃 t3-t10
    → 从 t2 重新开始
```

---

### 9.2.2 草稿模型 (Draft Model)

**草稿模型的作用**:
```
主模型和草稿模型的关系:
  主模型: "老师" - 准确但慢
  草稿模型: "助教" - 快速但不完美

草稿模型的要求:
  ✅ 速度快 (小模型,如 7B)
  ✅ 接受率尚可 (依任务而定)
  ✅ 与主模型架构相同或兼容
  ❌ 不需要完美 (可以接受一些错误)
```

**草稿模型的类型**:
```
类型 1: 小型号模型
  - Llama-2-7B 作为 Llama-2-70B 的草稿
  - 快速,但接受率较低

类型 2: 量化后的主模型
  - Llama-2-70B-INT4 作为 Llama-2-70B-FP16 的草稿
  - 快速,且架构完全相同

类型 3: 专门训练的草稿模型
  - 针对"预测主模型输出"优化
  - 接受率最高
  - 训练成本高
```

---

### 9.2.3 验证过程

**并行验证**:
```python
def verify_draft(main_model, prompt, draft_tokens):
    """
    主模型并行验证草稿 tokens

    Args:
        main_model: 主模型
        prompt: 输入 prompt (t0)
        draft_tokens: 草稿 tokens (t1, t2, ..., tn)

    Returns:
        verified_tokens: 验证通过的 tokens
        reject_position: 第一个不匹配的位置
    """
    # 拼接 prompt 和草稿 tokens
    full_input = [prompt] + draft_tokens

    # 主模型一次 forward pass
    outputs = main_model.generate(full_input)

    # 逐个比对
    for i in range(len(draft_tokens)):
        if outputs[i] == draft_tokens[i]:
            verified_tokens.append(draft_tokens[i])
        else:
            # 第 i 个 token 不匹配
            return verified_tokens, i

    # 全部匹配
    return verified_tokens, -1
```

**为什么可以并行验证?**
```
关键洞察: Attention 机制允许并行计算

传统生成:
  Token 0 → Token 1 → Token 2
  (串行,因为依赖关系)

投机采样验证:
  输入: [t0, t1, t2, ..., t10]
  → Attention 可以并行计算所有 tokens 的输出
  → 串行依赖关系体现在 attention mask 上
  → 一次 forward pass 验证所有 tokens
```

## 9.3 投机采样变体

**背景**：变体的差异，通常体现在三个维度：草稿模型从哪里来（小模型/量化主模型/专训草稿）、一次猜多少 token（speculation length）、验证策略如何组织（一次验收多个 token 的开销）。

**决策**：不要一上来就选“最复杂”的变体。先用最小实现测出接受率与端到端收益，再决定是否投入训练草稿模型或更复杂的验证策略。

### 9.3.1 Speculative Decoding

**原始投机采样方法**:
```
核心特点:
  - 使用小模型或量化模型作为草稿
  - 固定长度的 speculation (如 4-8 tokens)
  - 一次 forward pass 验证

优点:
  ✅ 实现简单
  ✅ 开销小
  ✅ 适用性广

缺点:
  ❌ 草稿模型可能接受率低
  ❌ 固定长度不够灵活
  ❌ 失败时浪费计算
```

**适用场景**:
```
✅ 长文本生成 (文章、代码)
✅ 高吞吐量场景
⚠️ 需要选择合适的草稿模型
```

---

### 9.3.2 Assisted Decoding

**核心思想**: 辅助解码
```
与 Speculative Decoding 类似
但强调"辅助"而非"投机"

区别:
  - Speculative Decoding: 强调"猜测"
  - Assisted Decoding: 强调"辅助验证"
```

---

### 9.3.3 Lookahead Decoding

**前瞻解码**:
```
核心特点:
  - 使用多个小草稿模型
  - 每个草稿模型"预测"不同长度的 tokens
  - 选择最长的正确序列

示例:
  草稿模型 1: 预测 4 tokens
  草稿模型 2: 预测 8 tokens
  草稿模型 3: 预测 12 tokens

  验证结果:
    草稿 1: 4/4 正确 ✅
    草稿 2: 6/8 正确 ❌
    草稿 3: 10/12 正确 ❌

  接受: 草稿 1 的 4 tokens
  最远验证: 草稿 3 的前 10 个 tokens
```

**优势**:
```
✅ 多个草稿模型提高接受率
✅ 自适应 speculation 长度
❌ 计算复杂度高
```

---

### 9.3.4 Eagle 系列: Eagle、Eagle 2、Eagle 3 ⭐

**背景**：你可以把 Eagle 系列理解为“专门为了投机采样训练的草稿模型家族”。与“直接拿一个小模型当草稿”相比，这类方法往往更强调接受率与可控性，从而在 decode-heavy 的负载上更容易形成正收益。

**决策**：把它当作一种“草稿训练范式”来理解更稳妥，不要把某个版本当成通用最优答案。你需要在自己的负载上回答三件事：

- 接受率是否稳定（平均值不够，要看分布与尾部）
- 草稿开销是否足够低（草稿太慢会把收益吃掉）
- 验证/回退是否会引入尾延迟风险（P95/P99 是否变差）

**落地**（建议顺序）：

1. 先用“可获得的草稿”（小模型或量化模型）跑通端到端与观测面
2. 再决定要不要投入训练专用草稿（目标是提高接受率/降低回退）
3. 最后用真实业务流量或近似回放做压测，决定是否上线

**踩坑**：

- 接受率在不同任务分布上差异巨大；需要“按 prompt 类别/长度分桶”看 acceptance
- 只看平均 TPS 容易误判；有些实现吞吐上升但尾延迟变差
- 验证开销的实现差异很大（kernel、batch、mask 处理都会影响）

**Eagle 系列演进（示例口径）**：

- **Eagle（早期）**：基础投机采样草稿训练，往往是固定 spec_len、单层草稿为主
- **Eagle 2（改进）**：更重视接受率与回退控制，可能引入更强的训练/蒸馏策略
- **Eagle 3（更工程化）**：更强调在推理框架里稳定集成与可观测性（但仍以你的压测结果为准）

### 9.3.5 方法对比

> 说明：下表为“工程对比口径”，用于帮助你在复杂度与潜在收益之间做初筛。具体收益必须以你的模型/负载/实现压测为准。

| 方法 | 草稿模型 | 收益潜力 | 尾延迟风险 | 工程复杂度 | 适合谁 |
|------|---------|---------|-----------|-----------|-------|
| **Speculative Decoding** | 小型/量化 | 中 | 中 | 中 | 先做原型验证的团队 |
| **Assisted Decoding** | 小型/量化 | 中 | 中 | 中 | 想要更稳妥的落地路径 |
| **Lookahead Decoding** | 多个小型 | 中-高 | 高 | 高 | 有研究/定制需求的团队 |
| **Eagle 类专训草稿** | 专训草稿 | 高 | 中 | 中-高 | 有压测体系、愿意训练草稿的团队 |

### 9.3.6 如何选择合适的变体

**决策树（工程化口径）**:
```
问题 1: 你现在的瓶颈是 decode-heavy 吗?
  否 → 先做 KV/调度/量化等基础优化（收益通常更稳）
  是 → 问题 2

问题 2: 你能拿到“兼容的草稿模型”吗?（小模型/量化模型/自训练）
  能 → 先用 Speculative Decoding 跑通端到端与观测面
  不能 → 先解决 tokenizer/架构兼容与工程接入

问题 3: 压测显示收益主要被“接受率/回退”限制吗?
  否 → 继续调参（spec_len、batch、验证策略）与优化实现
  是 → 评估“专训草稿模型”（Eagle 类范式）是否值得投入

问题 4: 你的 SLA 更看重尾延迟还是平均吞吐?
  尾延迟优先 → 需要更强的隔离与回退策略，否则容易把 P99 拉坏
  吞吐优先 → 更适合投入草稿训练与更激进的 spec_len
```

## 9.4 草稿模型选择

**背景**：投机采样“值不值”，很大程度取决于草稿模型。草稿模型越快不一定越好；更关键的是它能否在你的负载上保持高接受率。接受率不够高时，验证与回退会把收益吃掉。

**决策**：选草稿模型时优先看兼容性与可控性：

- 架构/Tokenizer 是否兼容（不兼容会引入额外映射与错误）
- 草稿分布是否接近主模型（越接近越容易被接受）
- 你愿不愿意为更高接受率付出训练成本（专训草稿）

### 9.4.1 小型号模型

**示例**:
```
主模型: Llama-2-70B
草稿模型: Llama-2-7B

优点:
  ✅ 速度快 (小模型 forward pass 快)
  ✅ 架构兼容 (都是 Llama 架构)
  ✅ 无需训练

缺点:
  ❌ 接受率可能较低 (强依赖任务/对齐/实现)
  ❌ 草稿质量差
  ❌ 加速比有限
```

**适用场景**:
```
✅ 快速原型验证
✅ 没有专门训练的草稿模型
❌ 追求极致性能
```

---

### 9.4.2 量化后的主模型

**示例**:
```
主模型: Llama-2-70B-FP16
草稿模型: Llama-2-70B-INT4

优点:
  ✅ 架构完全相同 (量化不改变架构)
  ✅ 接受率通常更高
  ✅ 无需训练

缺点:
  ❌ 量化后仍然可能较慢
  ❌ 显存占用仍较大

适用场景:
✅ 有足够显存
✅ 追求较好接受率
❌ 显存受限 (考虑小模型)
```

### 9.4.3 专门训练的草稿模型

**示例**:

主模型: Llama-2-70B

训练目标:
  预测主模型的输出

优点:

  ✅ 推理速度快
  ✅ 针对性优化

缺点:
  ❌ 训练成本高
  ❌ 实现复杂度高
  ❌ 需要大量训练数据
```

**训练流程**:

步骤 1: 收集训练数据

  - 收集 (prompt, output) 对

步骤 2: 训练草稿模型
  - 输入: prompt
  - 目标: 主模型的 output
  - 损失函数: Cross-entropy
  - 优化: AdamW

步骤 3: 验证
  - 测试 acceptance rate
  - 调优草稿模型
```

## 9.5 性能分析

**背景**：投机采样的性能讨论如果只说“数倍加速”，几乎一定会误导。你需要把收益拆成可解释的变量：接受率、speculation length、验证开销、回退概率、以及你的负载是 prefill-heavy 还是 decode-heavy。

**指标口径**：至少同时看：

- 接受率（acceptance rate）与其分布（平均值不够，尾部很重要）
- TTFT/TPOT 与 P95/P99（很多方案只改善平均值，尾部更差）
- 有效吞吐（扣除失败/重试/回退）与单位成本

### 9.5.1 理论加速比

投机采样的“理论加速”看起来很诱人，但要写对口径：它不是简单的 `1 + spec_len × accept_rate`。原因是：

- 验证并不是免费（主模型仍要做 forward，只是“验收多个 token”）
- 回退会产生额外成本（失败时要丢弃一段草稿，并从某个位置继续）
- 草稿模型自身也要跑（草稿越慢，收益越小）

一个更可解释、也更适合工程估算的写法是把时间拆开：

```
T_base ≈ N * t_main_decode

T_spec ≈ (#cycles) * (t_draft(m) + t_verify(m)) + t_fallback

其中:
  N: 输出 token 数
  m: speculation_len
  t_main_decode: 主模型单步 decode 平均耗时（你的负载压测得到）
  t_draft(m): 草稿生成 m 个 token 的耗时
  t_verify(m): 主模型一次验证 m 个 token 的耗时（受实现/批大小/内核影响）
  t_fallback: 回退带来的额外成本（与接受率分布有关）
```

**工程建议**：不要试图用一行公式“算出加速比”。更可靠的做法是做一个小规模回放压测，把 `acceptance rate 分布`、`verify 耗时`、`回退比例`测出来，再用同一套压测脚本扫 `m` 找最优点。

---

### 9.5.2 实际加速比影响因素

**关键因素**:

**1. Acceptance Rate (接受率)**:
```
Acceptance Rate = 验证通过的 tokens / 总投机 tokens

影响:
  - 接受率越高，回退越少，收益更容易出现
  - 但收益不会只由接受率决定，草稿开销与验证开销同样关键
  - 经验上，当接受率持续偏低（例如长期低于 50%）时，需要非常快的草稿与非常低的验证开销才能不亏

影响因素:
  - 草稿模型质量
  - 任务难度
  - 训练数据匹配度
```

**2. Speculation Length (投机长度)**:
```
投机长度: 每次投机多少 tokens

权衡:
  - 太短: 加速不明显
  - 太长: 失败风险高,浪费计算
  - 需要结合接受率与硬件约束调优
```

**3. 草稿模型速度**:
```
草稿模型越快 → 投机开销越小

理想情况:
  草稿模型时间 << 主模型时间
  投机开销可忽略
```

**4. 验证开销**:
```
验证并行化程度:
  - GPU 内存限制: 无法验证太长的序列
  - 并行度: 单次验证的 tokens 数量

实际考虑:
  - 投机长度受 GPU 内存限制
  - 需通过基准测试确定合理长度
```

---

### 9.5.3 什么时候投机采样有效

**最佳场景**:
```
✅ 长文本生成
   - 文章生成
   - 代码生成
   - 报告生成

✅ 高吞吐场景
   - 批量处理多个请求
   - 离线批处理

✅ 简单到中等难度任务
   - 故事续写
   - 代码补全
   - 翻译

✅ 草稿模型训练数据匹配
   - 草稿模型见过类似的输入
   - 领域对齐 (如代码生成模型用于代码生成)
```

**经验现象（用于帮助你找方向，不是保证值）**:
```
长输出、更稳定的分布:
  - 更容易从投机采样中受益（可被“批量验收”的 token 更多）

短输出、强格式约束:
  - 有时收益有限，甚至可能因回退/验证开销导致负收益

prefill-heavy 场景:
  - 投机采样主要优化 decode，对 prefill 占比高的负载往往帮助不大
```

---

### 9.5.4 什么时候会失败

**失败场景**:
```
❌ 短文本生成
  投机开销 > 收益
  示例: 短文本生成
  投机开销可能接近或超过收益

❌ 高难度任务
  草稿模型无法预测
  Acceptance rate 较低
  大量计算浪费

❌ 不匹配的领域
  草稿模型是通用模型,任务是专业领域 (如医学、法律)
  草稿模型质量差
  Acceptance rate 较低

❌ 资源受限
  GPU 显存不足以加载主模型 + 草稿模型
  无法并行验证

❌ 延迟敏感的实时应用
  需要稳定的延迟
  投机采样导致延迟抖动
```

**问题诊断**:
```
如果投机采样没有加速,检查:
  1. Acceptance rate 是否太低?
  2. Speculation length 是否合适?
  3. 草稿模型是否匹配任务?
  4. GPU 显存是否足够?
```

## 9.6 实战：把投机采样接到你的系统里

**背景**：实战落地的关键不是“把投机采样打开”，而是把它接入同一套线上指标、灰度与回滚机制里。投机采样最容易踩坑的地方是：平均吞吐变好，但尾延迟或稳定性变差；或者某一类请求加速明显，但另一类请求变慢。

**决策**：在写任何代码前，先做三件事（否则很容易“压测赢了，上线亏了”）：

1. 明确你的收益目标：吞吐优先还是尾延迟优先（两者的最优参数常常不同）
2. 明确你的负载形态：prefill-heavy 还是 decode-heavy（投机采样主要优化 decode）
3. 明确你的可回滚边界：接受率下滑、P99 变差、成本上升各自的报警阈值

**落地**：建议按“三步走”验证：

1. 离线基准：固定 prompt 分布与输出长度，测接受率与端到端收益
2. 小流量灰度：只放低风险流量，盯 P95/P99、超时率与失败重试率
3. 全量前评审：明确回滚条件（接受率阈值、尾延迟阈值、单位成本阈值）

### 9.6.1 配置投机采样

不同推理引擎的参数名会不同，但核心开关通常类似：

```yaml
speculative_decoding:
  enabled: true
  draft_model: /path/to/draft
  speculation_len: 8
  verify_mode: parallel
  fallback:
    enabled: true
    min_acceptance_rate: 0.5
    min_samples: 200
  observability:
    export_acceptance_rate: true
    export_fallback_rate: true
    export_verify_latency: true
```

**踩坑**：上线时一定要有“降级开关”。典型做法是当 `acceptance_rate` 连续低于阈值、或 `P99` 明显变差时，自动切回非投机路径。

---

### 9.6.2 选择合适的草稿模型

**选择标准（先粗分，再压测）**:
```
场景 1: 快速实验
  → 选择一个更小、且 tokenizer/架构兼容的模型做草稿（先跑通闭环）

场景 2: 生产落地（追求可控性）
  → 量化后的主模型或“更接近主模型分布”的草稿更容易形成正收益

场景 3: 追求极致收益
  → 评估训练专用草稿（Eagle 类范式），用更高接受率换取更稳的收益
```

**兼容性检查（示例口径）**:
```python
def check_compatibility(main_model, draft_model):
    """检查主模型和草稿模型的最低兼容条件（示例口径）"""
    assert main_model.arch == draft_model.arch, "架构不兼容"
    assert main_model.vocab_size == draft_model.vocab_size, "vocab 不匹配"
    assert main_model.tokenizer_id == draft_model.tokenizer_id, "tokenizer 不匹配"
    assert main_model.max_pos >= draft_model.max_pos, "max position 不匹配"
    print("✅ 兼容性检查通过")
```

---

### 9.6.3 性能基准测试

**指标口径**（不要只看一个 TPS）：
- `acceptance_rate` 分布（按场景/长度分桶）
- `TTFT/TPOT` 与 `P95/P99`
- `effective tokens/s`（扣除失败/回退）与单位成本
- `fallback_rate` 与 `verify_latency`（很多负收益来自这里）

**测试方法**：用同一组 prompts，分别跑“无投机”和“有投机”，并且固定输出长度/采样参数，避免噪声把结果淹没。

---

### 9.6.4 调优技巧

**技巧 1: 扫 speculation_len 找最优点**
```
spec_len 太小:
  - 验证开销摊不薄，收益有限

spec_len 太大:
  - 回退浪费增多，尾延迟风险变大
```

**技巧 2: 盯 acceptance 的“尾部”而不是均值**
```
平均 acceptance 高，但某一类请求 acceptance 很低:
  - 会把 P99 拉坏
  - 需要按桶隔离（只对某些流量启用投机）
```

**技巧 3: 把 batch 与调度一起看**
```
batch 更大 → 更高吞吐
但 batch 更大 → 可能改变接受率与尾延迟
```

## 9.7 实战：使用现成 draft checkpoint 的落地流程（以 Eagle 类草稿为例）

**背景**：与其一上来就投入训练草稿模型，很多团队会先尝试“现成的 draft checkpoint”。这条路线的价值是降低试错成本：你可以先把投机采样的收益与风险测清楚，再决定是否值得投入训练与深度定制。

**决策**：使用现成 checkpoint 前，先明确三件事：

- 你的主模型与 draft 是否兼容（架构/tokenizer/对齐方式）
- 你的系统是否具备灰度与回滚能力（避免尾延迟或质量回归扩散）
- 你是否能接受“对你的任务不一定最优”的现实（收益以压测为准）

### 9.7.1 什么是“现成 draft checkpoint”（以 Eagle 类草稿为例）

你可以把它理解为“一份可直接拿来做草稿的模型权重/配置”，它通常在训练时更强调：

- 更高的接受率（更像主模型输出分布）
- 更低的回退概率（减少浪费计算与尾延迟抖动）
- 更容易被推理引擎接入（可观测性与开关更完善）

它的价值不是“保证某个固定倍数的加速”，而是把你从“先训练再验证”变成“先验证再决定投入”。

---

### 9.7.2 现成草稿 vs 自训练草稿模型

| 维度 | 现成 draft checkpoint | 自训练草稿模型 |
|------|------------------------|----------------|
| **上手成本** | 低（更快跑通闭环） | 高（要数据、训练、评测） |
| **最优上限** | 不确定（未必贴合你的负载） | 更高（可针对你的业务分桶优化） |
| **可控性** | 低（参数/数据不可控） | 高（可控数据口径与训练目标） |
| **稳定性风险** | 中（依赖实现与兼容性） | 中-高（训练/评测口径不一致时更易翻车） |
| **维护成本** | 低-中 | 中-高（要持续跟业务分布漂移） |

---

### 9.7.3 在推理引擎中启用投机采样（示例）

不同引擎参数名不同，但通常都会暴露：草稿路径、spec_len、验证策略、以及可观测性指标。下面用伪代码表达“你需要哪些开关”：

```python
engine = launch_server(
    main_model="path/to/main/model",
    speculative={
        "enabled": True,
        "draft_model": "path/to/draft/checkpoint",
        "speculation_len": 8,
        "verify_mode": "parallel",
        "fallback": {"enabled": True, "min_acceptance_rate": 0.5},
    },
)
```

**踩坑**：把“投机采样开关”做成**按流量桶启用**，而不是全量开关。这样你可以先让“更适合投机”的流量吃到收益，把风险控制住。

### 9.7.4 性能基准测试

这一节更建议你把它当作“压测报告模板”，而不是固定结论。投机采样是否划算，必须用你的负载压测确认。

**压测环境（示例字段）**:
- GPU/驱动/CUDA
- 主模型与版本、最大上下文、采样参数
- 草稿模型与版本、speculation_len、验证策略
- 并发/批处理策略、流量形态（prefill-heavy 还是 decode-heavy）

**需要产出的指标口径**:
```
核心:
  - acceptance rate 分布（按 prompt 类别/长度分桶）
  - TTFT/TPOT 与 P95/P99（投机采样常见风险在尾部）
  - 吞吐（有效 tokens/s）与单位成本（$/1k tokens 或 GPU-min/1k tokens）

辅助:
  - 回退比例、回退平均长度
  - verify step 平均耗时、草稿 step 平均耗时
  - OOM/降级/fallback 次数
```

---

### 9.7.5 限制和注意事项

**兼容性与稳定性**:
```
仅在“架构/tokenizer/采样口径”兼容时更容易受益
不兼容会引入映射/截断/数值差异，导致接受率虚低或输出漂移
```

**资源与尾延迟**:
```
需要同时加载主模型与草稿模型，显存与带宽压力更大
若回退频繁，会把尾延迟放大（所以必须盯 P95/P99）
```

**适用边界**:
```
✅ 更适合长输出与吞吐优先的场景
⚠️ 短文本或强约束格式有时收益有限
❌ 极端延迟敏感场景要谨慎，优先做隔离与降级
```

### 9.7.6 Eagle 系列演进（示例口径）

把“版本”当成思路演进更有帮助：

- 早期：先把投机采样跑通（固定 spec_len、较简单的草稿训练）
- 中期：更重视接受率与回退控制（减少浪费与尾部抖动）
- 后期：更强调工程集成（可观测性、降级策略、与调度/批处理协同）

### 9.7.7 实战：自训练草稿模型的端到端流程 ⭐💡

把“专训草稿”（Eagle 类范式）落地到生产，最容易踩坑的不是训练本身，而是**数据口径、评测口径与上线口径**不一致。下面给一套更通用的端到端流程（框架/命令可能随版本变化，重点是结构与指标）。

**背景**：专训草稿模型的目标不是“做一个更聪明的小模型”，而是“更像主模型输出分布，且更便宜”。因此训练目标、数据采样与评估指标都要围绕 acceptance 与回退来设计。

**决策**：把训练目标拆成两个可验证问题：
- 接受率能不能稳定提升（按业务桶看）
- 草稿开销能不能足够低（不然提升的接受率也会被成本吃掉）

**落地流程（建议顺序）**:
```
步骤 1: 数据准备（对齐你的真实负载）
  - 采样真实线上 prompt（脱敏/过滤/去重）
  - 按场景分桶（短问答/长文/代码/强格式等）

步骤 2: 生成“监督信号”
  - 用主模型在你的采样参数下生成输出
  - 记录必要的中间信息（例如 logits/hidden states，视训练方法而定）

步骤 3: 训练草稿模型
  - 目标: 更高 acceptance、更低回退
  - 训练中就要做“在线模拟验收”，而不是只看 loss

步骤 4: 离线评测（必须贴近上线口径）
  - acceptance rate 分布（按桶）
  - TTFT/TPOT/P95/P99（对照无投机）
  - 单位成本（$/1k tokens）与失败/降级率

步骤 5: 灰度上线
  - 按流量/场景逐步放量
  - 配好快速降级开关（fallback 到非投机路径）
```

**踩坑清单**：
- 训练/评测使用的 tokenizer、采样参数与线上不一致，acceptance 会虚高或虚低
- 只在“干净数据”上训练，线上长尾 prompt 接受率崩溃
- 只盯平均 TPS，忽略 P99 被回退拖累（尤其是短请求）

## 🎯 总结

**关键要点**:
- 投机采样通过"猜测-验证"机制打破自回归生成的串行限制
- 收益是否出现主要取决于接受率分布、验证/回退开销与负载形态
- 草稿模型的选择（以及是否值得训练专用草稿）是性能的关键
- 长文本生成是投机采样的最佳场景
- 自训练草稿模型要把“数据口径/评测口径/上线口径”对齐，否则很难稳定受益
