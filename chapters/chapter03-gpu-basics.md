# 第3章：GPU基础

> **💰 商业动机**：推理成本的第一性约束来自硬件。相同模型与相同框架下，硬件选型与配置不当，推理成本和尾延迟常常会被放大到 3-5 倍。理解 GPU 的瓶颈（算力 vs 带宽 vs 显存）是后续所有优化（KV、调度、量化、投机采样、PD 分离）的基础。

## 简介

这一章不试图把 GPU 讲成硬件课，而是给你一套“够用且可行动”的思维模型。你需要能回答这些问题：

- 为什么 LLM 推理几乎一定要用 GPU？CPU 到底慢在哪？
- 同样是 80GB 显存，为什么两张卡的推理体验和成本会差很多？
- 你的瓶颈是算力、带宽、显存，还是“GPU 之外”（CPU/网络/队列）？
- 在不做任何 fancy 优化之前，如何做出不踩坑的 GPU 选型？

你会在本章建立三种能力：

- **直觉**：prefill 更像算力问题，decode 更像带宽问题（但要用指标验证）。
- **估算**：能在部署前算出“权重 + KV + 激活 + 开销”的显存账，避免 OOM 与盲目买卡。
- **诊断**：能用一组最小工具把“慢”变成可定位的问题。

## 3.1 CPU vs GPU：本质差异

### 3.1.1 类比：数学教授 vs 小学生团队

这个类比之所以常用，是因为它直观解释了“并行规模”：

- CPU 像一位数学教授：单线程强、分支与复杂逻辑强、延迟低，但并行规模有限。
- GPU 像一支小学生团队：单个线程不聪明，但可以同时做大量相同的简单计算。

LLM 推理里最核心的计算是矩阵乘（GEMM）与 attention 相关算子。它们的特点是：**同一种操作要重复做很多很多次**。这正是 GPU 擅长的模式。

### 3.1.2 并行计算 vs 串行计算

把推理过程粗略拆成两类工作：

- **大规模规则计算**：矩阵乘、向量加、归一化等，适合并行。
- **控制流与调度**：请求队列、token 级调度、内存分配、网络 IO 等，更像 CPU 的工作。

很多团队第一次上线推理服务时会踩一个坑：GPU 很强，但系统整体还是慢。原因是 GPU 只负责第一类工作，而第二类工作（CPU/IO/队列）也可能成为瓶颈。

因此你需要同时理解两件事：

- GPU 为什么快（并行 + 张量加速）。
- GPU 为什么“看起来没跑满”（瓶颈可能在带宽、显存、CPU 或调度）。

### 3.1.3 为什么 GPU 适合矩阵运算

LLM 推理里，绝大多数时间在做类似下面的计算：

```text
Y = XW
```

其中 `X` 是 token 的隐藏表示，`W` 是权重。无论你用什么模型结构，计算都会落到大量 GEMM 上。

GPU 在这类计算上占优的原因主要有三点：

- 计算单元多：大量并行线程同时工作。
- 内存带宽高：能更快把权重与中间结果从显存搬到计算单元附近。
- 张量核心（Tensor Cores）：专门针对矩阵乘优化的硬件路径（尤其是 FP16/BF16/FP8/INT8/INT4 等低精度）。

### 3.1.4 GPU 不适合的任务类型

GPU 并不是“万能加速器”。以下任务通常不适合放到 GPU 上（或需要非常谨慎）：

- 分支复杂、数据相关性强、并行度很低的逻辑（例如大量 if-else）。
- 小批量、小算子、频繁启动 kernel 的碎片化工作（启动开销显著）。
- 频繁在 CPU 与 GPU 之间搬数据（PCIe 传输开销会吃掉收益）。

这也是为什么推理优化经常是系统工程：你要尽量让 GPU 做它擅长的“密集计算”，同时减少 GPU 等 CPU、等 IO、等网络、等内存的时间。

## 3.2 GPU 架构详解：面向推理的思维模型

这一节的目标是：当你看到“显存 80GB、带宽 3TB/s、SM 数量、L2 Cache、NVLink”等参数时，你知道它们分别会影响推理的哪一部分指标（成本、吞吐、TTFT、TPOT、P95/P99）。

### 3.2.1 流式多处理器（SM）：GPU 的核心单元

可以把 GPU 理解成由很多个 SM 组成的“并行工厂”。每个 SM 里有：

- 计算单元（CUDA cores / Tensor cores）
- 寄存器文件（register file）
- 共享内存与 L1 cache（shared/L1）
- warp 调度器（决定哪些线程何时运行）

对推理来说，一个非常实用的经验是：

- **吞吐上限**通常与 SM 数量、张量核心能力有关。
- **尾延迟**往往与调度、内存层次结构与带宽有关。

但不要把“SM 越多越快”当作线性规律。因为当瓶颈在显存带宽时，SM 再多也只能等数据。

### 3.2.2 显存（VRAM）：容量 vs 带宽

显存有两个维度：

- **容量**：决定你能否放下权重、KV Cache、激活值与运行时开销。
- **带宽**：决定你能以多快的速度喂饱计算单元（尤其在 decode 阶段）。

推理里最常见的误判是：只看显存容量，不看显存带宽。

一个更贴近经营的理解方式是：

- 容量不够：你会 OOM，或者被迫降低 batch/并发/上下文，直接限制业务规模。
- 带宽不够：你会 TPOT 变慢、P95/P99 抖动、为保 SLA 多买冗余卡，导致单位成本上升。

### 3.2.3 内存层次结构：寄存器 / L1 / L2 / HBM(GDDR)

GPU 访问数据的速度是分层的：

- 寄存器：最快，但很小。
- shared/L1：很快，适合 block 内复用。
- L2：跨 SM 共享，容量更大。
- 显存（HBM/GDDR）：容量最大、带宽高，但相对 cache 仍然更慢。

对推理来说，cache（尤其 L2）变大往往能改善部分工作负载，因为：

- 更多热点数据可以停留在更靠近计算的层级。
- 更少访问显存，减少带宽压力。

但它不是“万能加速”。你要用 profiling 与端到端指标验证收益。

### 3.2.4 带宽：推理的真正瓶颈（很多时候）

一个非常重要的直觉：

- **Prefill** 更容易是 compute-bound（计算密集）。
- **Decode** 更容易是 memory-bound（带宽/访存密集）。

你可以用 Roofline 模型建立上限意识：

```text
性能上限 = min( 计算峰值 , 带宽峰值 × 算术强度 )

算术强度 = FLOPs / Bytes
```

当某个 kernel 的算术强度很低时，性能上限由带宽决定。很多 decode 阶段的工作正是如此：每一步产生一个 token，但要访问大量历史 KV 与权重相关数据，算术强度并不高。

因此，推理优化里大量“看起来像软件技巧”的东西（KV 组织、paged/chunked、连续批处理、前缀缓存）本质上都是在做同一件事：**减少无效访存与提升有效带宽利用率**。

### 3.2.5 PCIe / NVLink：GPU 与 CPU 的桥梁

当数据需要在 CPU 与 GPU 之间来回搬运时，互连就会成为瓶颈。

几个常见场景：

- 模型权重不在显存：需要从主机内存或磁盘加载（启动慢、抖动大）。
- KV Cache 跨设备/跨节点传输：PD 分离、分布式服务时会遇到。
- 多卡并行：张量并行、流水并行时通信影响尾延迟。

工程建议：

- 尽量减少“每 token 跨 PCIe 传输”的设计。
- 真要做跨设备方案，先把“传输成本”作为一等指标测出来（否则你在黑箱里优化）。

### 3.2.6 Tensor Cores 与 Transformer Engine：低精度的主航道

推理优化里最常见的“第一刀”是量化或低精度，因为它同时影响：

- 权重占用（显存容量）
- 访存量（带宽压力）
- 张量核心吞吐（计算上限）

你不需要在这一章就掌握所有精度细节，但要有一个基本判断：

- 精度越低，理论吞吐与有效带宽越高。
- 精度越低，质量风险越大，越需要任务级回归集与灰度策略。

### 3.2.7 SIMT 执行模型：Warp、Block 与 Occupancy

GPU 的并行执行模型常被概括为 SIMT：一条指令由一组线程（warp，通常 32 个）一起执行。

这带来两个对推理很重要的工程含义：

- 分支发散会降低效率（同一个 warp 里走不同分支，会串行化）。
- “看起来很忙”的 GPU 也可能有效吞吐不高，因为 occupancy 不足或在等内存。

你不需要自己写 CUDA kernel，但要能读懂 profiler 的关键词：warp、stall、memory throughput、occupancy，这些会在后面章节频繁出现。

## 3.3 显存计算公式：先把账算清楚

部署前做显存预算，是降低试错成本的最有效动作之一。常见拆分如下：

```text
总显存 ≈ 权重 + KV Cache + 激活值 + 运行时开销
```

训练还需要梯度与优化器状态，显存需求会显著更高；本章只讨论推理。

### 3.3.1 模型权重计算

权重占用的下界很好算：

```text
权重显存 ≈ 参数量 × 每参数字节数
```

粗略参考：

- FP16/BF16：2 bytes
- INT8/FP8：1 byte（实现不同会有额外开销）
- INT4：0.5 byte（通常还会有 group scale 等元数据）

例：`8B` 参数在 FP16 下约 `16GB`；`70B` 参数在 FP16 下约 `140GB`，单卡显然放不下，需要量化或多卡并行。

### 3.3.2 KV Cache 显存占用

一个更通用、也更容易写进表格的 KV 估算公式是：

```text
KV bytes
  = 2 * num_layers * seq_len * batch
    * kv_heads * head_dim * bytes_per_element
```

解释：

- `2` 表示 K 和 V 两份。
- `seq_len` 是你要保留的历史长度（对话越长越大）。
- `batch` 在推理服务里更接近“并发活跃请求数”（不是训练的 batch 概念）。
- 如果是 GQA/MQA，则 `kv_heads < attn_heads`，KV 会显著变小。

非常重要的提醒：

- KV 增长是线性的：`seq_len` 翻倍，KV 翻倍；并发翻倍，KV 翻倍。
- 在长上下文与多轮对话场景，KV 往往会超过激活值，成为显存主导项。

### 3.3.3 激活值显存

推理阶段的激活值通常显著小于训练，但在某些实现与较大 batch 下仍然不可忽略。工程上更稳妥的做法是：

- 先按权重 + KV 做下界估算。
- 再预留 10%-30% 做激活与运行时缓冲（按框架、并行方式与工作负载调整）。

### 3.3.4 CUDA 运行时开销

CUDA 上下文、框架缓存、分配器碎片、多卡通信缓冲都会吃掉显存。你在 `nvidia-smi` 里看到的“空载占用”就是它的一部分。

如果你把预算算到 99% 才“刚好放下”，上线后几乎一定会 OOM 或抖动。

### 3.3.5 实战计算：Llama-3-8B 需要多少显存（示例）

一个可操作的估算流程（示例口径，不代表某个模型的官方配置）：

1. 权重：`8B` × `2 bytes` ≈ `16GB`
2. 预留激活与开销：先预留 `3-5GB`（按实现调整）
3. 剩余给 KV：`24GB` 卡上可能还有 `3-5GB` 的 KV 空间

你会得到一个非常工程化的结论：**24GB 卡能跑 8B，但并发与上下文会很快成为瓶颈**。这不是“卡不行”，而是你需要更明确的目标：更低精度、更短上下文、更小并发，或者换更合适的硬件/架构。

### 3.3.6 实战计算：Llama-3-70B 如何放得下（方法论）

70B 在 FP16 下单卡放不下，这是一个很常见的“逼你选路”的节点。通常有四种路径：

- 量化：INT8/INT4，把权重压到可承受范围（需要回归评测与质量保护）。
- 多卡并行：张量并行/流水并行（需要更复杂的通信与运维）。
- 更小模型 + 路由：用分级模型解决大多数请求，强模型只处理高价值请求。
- 缓存与复用：如果大量请求共享前缀或模板，上下文复用能显著降低成本。

这四条路径都不是“GPU 课”的内容，但它们决定了你后续章节要重点学什么。

## 3.4 GPU 性能监控：把黑箱变成仪表盘

这一节的目标是：你能用最小工具回答三个问题：

- 我现在是不是 OOM 风险？
- GPU 真在忙吗？忙的是计算还是访存？
- 为什么 P95/P99 抖动？是 GPU 还是队列/CPU/网络？

### 3.4.1 `nvidia-smi` 详解（够用版）

最基础的命令：

```bash
nvidia-smi
```

你需要重点看三类信息：

- `Memory-Usage`：容量占用（是否接近上限，是否在增长）
- `GPU-Util`：计算利用率（高不等于快，低不等于没用）
- `Pwr:Usage/Cap`：功耗（功耗低但占用高，常见于访存/IO/CPU 受限）

### 3.4.2 持续监控工具

当你在压测或线上排障时，建议用持续采样而不是偶尔 `nvidia-smi`：

```bash
watch -n 1 nvidia-smi
```

更进阶一点，你可以把关键字段以 CSV 输出，便于落盘与对齐时间线（不同版本字段略有差异，可自行调整 query 项）。

### 3.4.3 Python 监控：`pynvml`

当你要把 GPU 指标接到你的观测系统里（Prometheus/自建日志），NVML 是常用入口。

```python
import pynvml

pynvml.nvmlInit()
handle = pynvml.nvmlDeviceGetHandleByIndex(0)

mem = pynvml.nvmlDeviceGetMemoryInfo(handle)
util = pynvml.nvmlDeviceGetUtilizationRates(handle)

print(f"mem_used_gb={mem.used / 1024**3:.2f}")
print(f"mem_total_gb={mem.total / 1024**3:.2f}")
print(f"gpu_util={util.gpu}")
print(f"mem_util={util.memory}")
```

### 3.4.4 性能计数器（知道它存在即可）

当经验法则不够用时，你会需要 profiler（例如 Nsight Systems/Compute）去回答：

- GPU 在等什么（memory stall、warp stall、launch overhead）？
- 哪些 kernel 最贵？
- 带宽到底有没有吃满？还是被访问模式拖垮？

这一章不展开 profiler 的细节，但你需要知道：当你要进入“最后 2 倍”的优化时，它是绕不过去的工具。

## 3.5 性能瓶颈诊断：三大瓶颈类型

### 3.5.1 三大瓶颈类型（够用版）

你可以先用一个简化分类来做第一轮判断：

1. **显存受限（Capacity-bound）**
   - 症状：OOM、频繁重启、batch/并发上不去、上下文稍长就崩。
   - 常见手段：量化、降低并发、缩短上下文、改 KV 精度、改模型结构或并行方案。

2. **带宽/访存受限（Memory-bound）**
   - 症状：`GPU-Util` 不高但 `TPOT` 很慢；功耗不高；尾延迟抖动明显；长上下文更差。
   - 常见手段：KV 管理与复用、连续批处理、减少无效 token、优化 attention 内核、提升有效带宽利用率。

3. **计算受限（Compute-bound）**
   - 症状：`GPU-Util` 高且稳定；功耗高；TPOT 随算力提升明显改善。
   - 常见手段：更强 GPU、低精度（提高张量核心吞吐）、更高效的 kernel。

还有一个“第四类”你必须牢记：**GPU 之外的瓶颈**（CPU/网络/队列）。

如果你的 GPU 利用率很低，但 CPU 单核打满、或者请求排队严重，那你优化 GPU 的收益会很有限。

### 3.5.2 诊断流程图（第一轮）

```text
先看：是否 OOM / 逼近显存上限？
  ├─ 是：显存受限 → 先解决容量（量化/并发/上下文/KV）
  └─ 否：看 TPOT 变慢时 GPU-Util 与功耗
        ├─ GPU-Util 高 + 功耗高：更像计算受限
        ├─ GPU-Util 中低 + 功耗不高：更像带宽/访存受限
        └─ GPU-Util 很低：检查 CPU/网络/队列/数据搬运
```

这个流程图不是结论，只是帮你快速把排查空间从“无穷大”缩到“三四个方向”。

### 3.5.3 实战案例：分析真实推理瓶颈（怎么做）

一个可复用的排障套路（你后面章节会反复用到）：

1. 先固定一个可复现负载：同一模型、同一上下文长度、同一并发与输出长度。
2. 同时采集三条时间线：请求延迟分位（P50/P95/P99）、GPU 指标（util/mem/power）、队列长度/CPU 利用率。
3. 做一次“最小改动 A/B”：只改一个变量（例如量化开关、batch 策略、KV 精度、上下文长度）。
4. 看指标变化：如果变化与你预期相反，优先怀疑“瓶颈类型判断错了”。

## 3.6 常见 GPU 规格对比：怎么选，不怎么吵

这一节不给你“唯一答案”，只给你一个避免踩坑的选型框架。

### 3.6.1 消费级 GPU（RTX 系列）：适合什么

典型优势：

- 单卡性价比高（尤其在你不需要 ECC、不需要严格 SLA 的阶段）。
- 开发调试方便，适合原型验证与小模型部署。

典型限制：

- 显存容量相对有限（长上下文/高并发很容易卡在 KV）。
- 稳定性与运维特性（ECC、长时间满载、集群管理）通常不如数据中心卡。

### 3.6.2 数据中心 GPU（A100/H100 等）：贵在哪

你买到的不仅是“更快”，还包括：

- 更高显存带宽（常直接影响 decode TPOT 与尾延迟）。
- 更强的互连与集群形态（多卡并行、跨节点通信）。
- 更好的可运维性与稳定性（对生产 SLA 更友好）。

### 3.6.3 云 GPU 选择指南（最实用的一条）

云上选 GPU，建议按这个顺序问问题：

1. 你的上限约束是什么？容量、带宽、算力，还是稳定性？
2. 你的流量是什么形态？短 prompt 还是长上下文？峰值明显吗？并发要多高？
3. 你愿意用哪些软件杠杆？量化、缓存、路由、PD 分离、多卡并行？

不要先问“哪张卡最好”。先问“我的瓶颈是什么”，答案会更清晰。

### 3.6.4 性价比分析：不要只看 $/TFLOPS

推理的经营指标往往更接近：

- $/有效回答（含失败/重试/兜底）
- $/并发会话（在目标 P95/P99 下）
- $/每任务（Agent 场景）

因此你需要的不是“纸面算力最大”，而是“在你的真实负载下单位成本最低且 SLA 可控”。

## 3.7 NVIDIA 架构演进：从 A100 到 B200（为什么你需要关心）

这一节的目的不是背参数，而是理解“每代 GPU 的改进会把你的瓶颈推向哪里”。通常会看到三类变化：

- 更强张量核心：低精度更快 → 量化/低精度收益更大。
- 更高显存带宽与更大 cache：decode 更受益 → 长上下文与并发更容易做稳。
- 更强互连：多卡并行与拆分架构更可行 → 系统形态会更复杂但潜力更大。

工程建议：

- 如果你的瓶颈在 decode（带宽/尾延迟），优先关注带宽与内存层次结构，而不是只看峰值 FLOPS。
- 如果你计划做多卡并行或 PD 分离，互连与通信形态会决定上限与复杂度。

本书更深入的参考资料与链接汇总在 `docs/refs.md`，你可以在需要做硬件规划时再深入读。

## 常见误区

- 误区1：`VRAM` 越多越好。
  容量不够确实会直接失败，但容量够用后，推理更常见的瓶颈是带宽与尾延迟治理。
- 误区2：batch 越大吞吐一定越高。
  在在线服务里，请求长度分布与尾延迟约束会改变最优 batch；盲目增大 batch 可能让 P95/P99 变差，反而提高冗余成本。
- 误区3：`GPU-Util` 低就表示浪费。
  你可能是 memory-bound、CPU-bound 或 queue-bound；util 只是信号，不是结论。
- 误区4：温度高就是负载高。
  温度更多反映散热与功耗状态，不等价于有效吞吐。
