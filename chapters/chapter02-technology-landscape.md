# 第2章 技术全景：我们在 2025 年看到的几个关键趋势

> **我们这一章想做的事**：帮你建立一个”地图”——当你面对一个优化需求时，知道该往哪个方向走、以及为什么要往那个方向走。

我们在 2024-2025 年观察到一个现象：**很多团队在推理优化上花了很多时间，但最后发现方向错了**。这不是他们不努力，而是因为：

- 有人追热点，看到 “PD 分离” 火就想去试，结果自己的业务场景根本不需要
- 有人闷头调参数，调了几个月发现瓶颈根本不在那里
- 有人炫技，用了最新框架，最后运维成本高到哭

这一章我们不只讲”有什么技术”，更想帮你建立**判断框架**——当你面对一个新概念时，怎样判断它是否值得投入、以及什么时候该投入。

## 2.1 技术趋势概览

趋势不是“答案”，但它能帮你更快地判断：哪些能力会变成标配，哪些只是阶段性技巧；哪些优化会变成长期壁垒，哪些只是一次性红利。

### 2.1.1 DeepSeek V3：MoE 范式的代表性案例

DeepSeek V3 的讨论热度，很大程度来自一个关键命题：**当模型规模持续扩大时，能否把推理成本压回可承受区间？**

从概念上看，MoE（Mixture of Experts）的核心承诺是：总参数很大，但每次只激活一小部分参数，从而把“能力上限”和“计算成本”部分解耦。

**背景**：密集模型（Dense）在推理端的瓶颈经常不是“算不动”，而是“算得起吗、扛得住吗”。

- 大模型意味着更高的显存压力、更长的 prefill、更贵的并发承载成本。
- 业务一旦进入高频调用，单位 token 的差异会被总调用量放大成经营级差异。

**决策**：什么时候你该认真评估 MoE？

- 你的业务确实需要更高能力，但预算又不支持一直用更贵的密集模型。
- 你可以接受更复杂的服务系统（路由、负载均衡、分布式与容灾）。
- 你有足够强的观测与回归体系，因为 MoE 的不稳定性更容易体现在尾延迟与长尾质量上。

**落地**：从“能跑”到“跑得稳”，MoE 推理的工程关注点常集中在四件事：

- 路由与负载均衡：不同专家被选择的频率并不均匀，容易出现热点专家。
- 专家并行与通信：跨卡/跨节点时，通信开销与同步方式会直接影响尾延迟。
- 批处理与调度：MoE 的有效 batch 往往更依赖“同路由样本聚类”，否则吞吐会掉。
- 故障与退化：专家不可用时如何降级？如何避免局部故障放大成全局抖动？

**踩坑**：MoE 最常见的坑，不是“理论算力更省”，而是“系统真实交付更难”。

- 吞吐可能提升，但 P95/P99 反而变差（热点专家、通信抖动、队列不平衡）。
- 监控粒度不足时，你只能看到“整体慢了”，但不知道慢在哪个专家/哪条路由。
- 质量回归更隐蔽：某些任务被路由到不合适的专家，表现是“偶发错误”而不是整体下降。

**指标口径**：评估 MoE 是否值得做，不要只看 tokens/s。

- 成本：单位 token 成本、单位有效回答成本（含失败/重试/兜底）。
- 性能：TTFT（首 token 延迟）、TPOT（每 token 时间）、P95/P99 延迟。
- 稳定性：专家负载方差（热点程度）、路由分布漂移、尾延迟抖动来源（通信/排队/计算）。
- 质量：任务级回归集上的一致性、长尾任务错误率、引用/工具调用的成功率。

结论并不需要是“MoE 一定更好”。更有用的结论是：**MoE 把优化重点从单卡算力，推向了分布式系统与调度能力**。如果你没有相应的系统能力，MoE 可能会把成本从算力转移到工程复杂度与稳定性上。

### 2.1.2 PD 分离（Prefill-Decode 分离）：一种重要的服务模式

PD 分离（Prefill/Decode Disaggregation）之所以成为趋势，是因为它对应了推理的一个结构性事实：**prefill 和 decode 是两种完全不同的工作负载**。

**背景**：同一张 GPU 同时做 prefill 和 decode，常见症状是“平均利用率不低，但体验很差”：

- prefill 更像计算密集型：大的矩阵乘、attention，适合吞吐型 batch。
- decode 更像带宽/延迟敏感型：每步都要访问 KV Cache，且请求之间彼此干扰大。

当两者混在一起时：

- 峰值流量来临，会互相抢资源，尾延迟抖动更明显。
- 你为了保 SLA 被迫加冗余，单位成本上升。

**决策**：什么时候值得做 PD 分离？

- 你的业务对 P95/P99 很敏感（在线交互、客服、搜索问答、Agent 工具链）。
- 你的流量形态比较复杂：短 prompt 与长上下文混杂、峰值明显、并发波动大。
- 你已经做过基础优化（批处理、KV 管理、量化等），仍然被尾延迟或资源错配困住。

**落地**：PD 分离的最小可行版本不是“把系统拆成两半”，而是先把边界定义清楚：

- 哪些请求可以延后 prefill？
- KV Cache 在 prefill 与 decode 之间怎么传？（同机共享、同节点共享、跨网络传输）
- decode 池如何扩缩容？如何避免排队导致的雪崩？

一个典型的概念图如下：

```text
Request
  ├─ Prefill Pool  (吞吐优先，尽量批处理)
  │    └─ 输出：KV Cache + 首 token
  └─ Decode Pool   (尾延迟优先，尽量稳定)
       └─ 输出：后续 tokens
```

**踩坑**：PD 分离不是“必然更快”，它经常把瓶颈从 GPU 内部挪到系统边界。

- KV 传输与序列化开销可能吃掉收益，尤其是跨网络时。
- 逻辑复杂度上升：两套调度、两套扩缩容策略、更多故障模式。
- 如果观测不全，你会在“prefill 很快但整体很慢”的状态里迷失。

**指标口径**：PD 分离要用“体验指标 + 资源指标”一起评估。

- 体验：TTFT、TPOT、P95/P99、超时率、降级率。
- 资源：prefill/decode 的 GPU 利用率、显存占用、KV 传输带宽、队列长度。
- 经营：单位请求成本、峰值冗余比例（为保 SLA 多买的那部分）。

一句话总结：PD 分离的价值往往不是把 P50 拉得多快，而是把 P95/P99 拉得更稳，进而降低你为了保 SLA 被迫支付的“冗余税”。

### 2.1.3 RL Info 的兴起：推理与训练的边界正在消失

很多团队对 RL（RLHF/RLAIF/各种后训练）仍然用“训练视角”理解：它是训练团队的事。但在 2025 之后，越来越多的 RL 系统会把推理基础设施推到前台，因为 RL 的核心资源消耗之一是**大规模 rollout（采样/生成）**，本质上就是“以训练目的组织的推理”。

**背景**：当你做后训练时，推理不再只是面向用户的在线服务，还包括：

- 生成大量候选回答、轨迹、工具调用序列（rollout）。
- 做评估、打分、筛选（reward model 或规则评估）。
- 做数据回流与再训练（pipeline 化）。

这会带来一个结构性变化：你的集群不再是单一工作负载，而是混合负载（推理 + 数据处理 + 训练 + 评估）。

**决策**：什么时候你需要把“RL 视角”纳入推理设计？

- 你计划做持续迭代的后训练，而不是一次性微调。
- 你的业务需要领域对齐（合规、风控、专业回答）且对质量敏感。
- 你已经发现“模型能力不稳定”而不仅是“系统性能不够”。

**落地**：把 RL 与推理放在同一张图里，你会发现关键问题变成：

- 你如何把 rollout 任务排进集群，不挤爆线上服务？
- 你如何在混合负载下做隔离、配额与优先级？
- 你如何度量“每单位成本带来的质量提升”，而不是只看训练 loss？

**踩坑**：RL 系统常见的坑在于“推理层面的隐性成本”：

- rollout 的上下文通常更长、工具链更复杂，KV 与带宽瓶颈更明显。
- 评估与打分本身也消耗推理资源，导致你以为是在训练，实际在烧推理。
- 缺少统一指标时，优化很容易变成“某个团队的局部最优”。

**指标口径**：把 RL 相关推理当成生产系统来做。

- 产能：rollout tokens/s、每小时可生成的有效样本数、失败重试率。
- 成本：每条轨迹成本、每个有效样本成本、离线任务对线上 SLA 的影响成本。
- 质量：回归集提升幅度、偏好一致性、稳定性（方差）而不只是均值。

这就是 RL Info 的核心影响：它迫使你把推理当成“通用智能生产线”，而不是“一个在线接口”。

### 2.1.4 Agent 与多模态的爆发：上下文工程成为新瓶颈

当系统从“单轮问答”变成“多步任务”（检索、工具调用、计划与执行），推理系统的瓶颈会发生明显迁移：**上下文与工作流**会比模型本身更先决定体验和成本。

**背景**：Agent/多模态系统天然倾向于：

- 更长上下文（历史对话、计划、工具结果、引用证据）。
- 更多 token（多轮澄清、反思、重试、工具调用返回）。
- 更多外部依赖（检索、数据库、执行环境）。

这会把推理成本从“每次问答成本”变成“每个任务成本”，而且任务成本的方差会非常大。

**决策**：你需要把 Agent/多模态当成推理设计的一等公民吗？

- 你的产品不是“回答一个问题”，而是“完成一个任务”（写一份报告、修一个 bug、做一次排障、生成一份合规审阅）。
- 你的流量以长对话/长文档/多工具调用为主。
- 你的失败成本高（错误工具调用、错误引用、越权访问、提示注入）。

**落地**：Agent 系统的推理优化常常先从“控制上下文”开始，而不是先上更大的 GPU。

- 约束上下文：把“必须在上下文里”的内容和“可以检索得到”的内容分开。
- 规范化与缓存：相似系统提示词、相似模板、相似工具说明应该尽量复用。
- 分级模型路由：高风险/高价值步骤用强模型，其他步骤用轻模型。
- 失败处理：把重试策略、降级策略写成确定性的流程，而不是靠随机采样碰运气。

**踩坑**：Agent 系统里最贵的往往不是一次推理，而是“反复失败”。

- 工具调用失败导致多次重试，token 成本飙升。
- 上下文膨胀导致 decode 变慢，尾延迟恶化，用户感知变差。
- 缺少权限与审计导致上线受阻，或者上线后出现安全事故。

**指标口径**：Agent 的关键单位不再是“每 token”，而是“每任务”。

- 体验：任务完成时间、任务成功率、重试次数分布、P95/P99 任务耗时。
- 成本：每任务 token、每任务推理成本、工具调用失败带来的额外成本。
- 质量：引用正确率、工具调用正确率、越权/注入防护命中率、人工介入比例。

这也是为什么近两年“Context Engineering”成为热词：它不是写提示词，而是在做“可控的上下文供给与成本治理”。

### 2.1.5 从 SPMD 到 MPMD：推理栈越来越深，系统越来越像“工厂”

你可以把过去的主流训练范式理解为 SPMD：大量相同结构的 worker，跑相同类型的任务（预训练），调度策略相对单一。

但随着 RL、Agent、多模态与 PD 分离等趋势叠加，系统正在走向 MPMD：不同类型的 worker 并存，任务类型多样，资源约束差异巨大。

**背景**：一个“看起来只是推理服务”的系统，实际上常常包含：

- 网关与路由：鉴权、配额、模型选择、灰度。
- Prefill 池与 Decode 池：不同优化目标，不同扩缩容策略。
- KV/缓存层：前缀缓存、会话缓存、跨请求复用策略。
- 工具与检索：embedding、向量库、搜索、执行环境。
- 评测与观测：端到端指标、回归集、事故复盘与回滚。

**决策**：你要避免两种极端：

- 过早复杂化：刚上线就把所有趋势都集成，系统不可控。
- 长期简化：流量上来后仍然用“单体推理服务”，被尾延迟与成本逼到墙角。

更稳妥的策略是：用 `2.2` 的方法先建立优先级，按“最小可行闭环”逐步增加系统能力。

## 2.2 五大优化方向速览

这一节的目标不是让你记住所有技术名词，而是让你形成一个可执行的方法：**先识别约束，再选择方向，然后用指标闭环验证。**

### 2.2.1 快速评估矩阵

先回答一个问题：你的主要约束是什么？不同约束决定了你应该先动哪一类杠杆。

下面给一个非常实用的“第一轮评估矩阵”（你可以把它当成开会用的一张表）：

| 你最在意什么 | 典型症状 | 第一优先杠杆 | 第二优先杠杆 |
| --- | --- | --- | --- |
| 成本（$/req） | 账单失控、毛利下滑 | 模型优化（量化/蒸馏/路由） | 缓存与复用 |
| 延迟（P95/P99） | 交互卡顿、超时、抖动 | 系统架构（PD/拆分/隔离） | 调度优化 |
| 吞吐（QPS/tokens/s） | 峰值扛不住、排队 | 调度优化（连续批处理） | 推理引擎优化 |
| 稳定性（SLA） | 事故频繁、不可恢复 | 架构与运维（降级/回滚/观测） | 引擎与缓存 |
| 质量（正确率/一致性） | “偶发错误”多 | 评测与工作流（RAG/工具/边界） | 模型与路由 |

接下来，把五大方向放回这张表里，你会更容易理解它们的职责边界。

**方向1：系统架构优化（System Architecture）**

- 关注点：整体架构设计、硬件选型、工作流拆分、隔离与弹性。
- 典型技术：PD 分离、网关与路由、服务拆分、混部与容灾、多模型部署。
- 适用场景：一切生产系统，尤其是 P95/P99 与稳定性优先的系统。
- 常见误区：为了“理论最优”把系统拆得过细，反而不可运维。

**方向2：推理引擎优化（Inference Engine）**

- 关注点：框架选择与配置、内核效率、算子融合与内存管理。
- 典型技术：vLLM/SGLang/TensorRT-LLM，attention 内核优化、paged/chunked KV。
- 适用场景：当你已经有稳定业务流量，并且需要可复制的性能提升。
- 常见误区：只看 microbench，不看端到端（网络、排队、缓存命中、失败重试）。

> **选型建议**：
> - 通用场景 → vLLM（生态最全，文档完善）
> - 追求极致性能 + 有 SGLang 经验 → SGLang
> - 生产环境 + NVIDIA 硬件 → TensorRT-LLM（优化最深，但生态封闭）
> - 快速原型 + 灵活实验 → vLLM 或 SGLang

> **基准测试参考** (Llama-3.1-8B, A100-80GB, batch=32, input=512/output=128)：
> - vLLM: ~180 tokens/s
> - SGLang: ~200 tokens/s (+11%)
> - TensorRT-LLM: ~190 tokens/s (+6%)
>
> 数据来源：Chatbot Arena 基准测试（2025 Q1），实际性能因场景而异，建议在真实流量下验证。

**方向3：模型优化（Model Optimization）**

- 关注点：压缩、结构、路由与能力-成本权衡。
- 典型技术：量化（FP16→INT8/INT4/FP8）、蒸馏、剪枝、MoE、分级模型路由。
- 适用场景：成本敏感、显存受限、或者需要在更多硬件上部署。
- 常见误区：只盯压缩率，不做任务级回归，导致“便宜但不可用”。

**方向4：请求调度优化（Request Scheduling）**

- 关注点：如何组织请求与 token 的计算顺序，提升 GPU 有效利用率。
- 典型技术：连续批处理、优先级队列、chunking、不同流量形态的隔离。
- 适用场景：高并发、流量波动大、吞吐是硬约束的场景。
- 常见误区：把 batch 做大但忽略尾延迟，导致体验崩盘。

**方向5：缓存与复用（Caching & Reuse）**

- 关注点：避免重复计算，让相似请求共享计算结果。
- 典型技术：前缀缓存、会话缓存、模板化上下文复用、跨请求 token 复用。
- 适用场景：系统提示词稳定、请求模板化、或者对话型场景。
- 常见误区：只追求命中率而忽略一致性与失效策略，导致“缓存污染”与质量回归。

到这里你不需要立刻选“最强技术”，你需要选“最适合你约束的第一刀”。

### 2.2.2 技术选型决策树

下面这棵决策树用于“第一轮选择”。它的目的不是给你唯一答案，而是把讨论从“观点争论”变成“指标驱动”。

```text
你的主要约束是什么？
├─ 成本敏感（预算有限、毛利压力）
│  ├─ 上下文很长/对话多？
│  │   ├─ 先做：缓存与复用（前缀/会话/模板）
│  │   └─ 再做：模型优化（量化、分级路由）
│  └─ 上下文不长但调用量大？
│      ├─ 先做：模型优化（量化通常是第一刀）
│      └─ 再做：调度优化（连续批处理）
│
├─ 延迟敏感（实时交互、P95/P99 要求）
│  ├─ 抖动明显、峰值时更差？
│  │   ├─ 先做：系统架构（隔离、PD 分离、弹性）
│  │   └─ 再做：调度（优先级、队列与隔离）
│  └─ 平均延迟高但抖动不明显？
│      ├─ 先做：推理引擎（内核/配置/显存策略）
│      └─ 再做：模型优化（量化/更小模型）
│
├─ 吞吐敏感（峰值扛不住、排队严重）
│  ├─ GPU 利用率低？
│  │   ├─ 先做：调度（连续批处理、chunking）
│  │   └─ 再做：引擎（减少 CPU 开销、提升并行）
│  └─ GPU 利用率高但仍扛不住？
│      ├─ 先做：模型优化（量化/更小模型/路由）
│      └─ 再做：系统架构（横向扩展、拆分）
│
└─ 质量敏感（错一次代价很高）
   ├─ 先做：评测与边界（回归集、灰度、兜底策略）
   ├─ 再做：工作流（检索、工具调用、引用约束）
   └─ 最后做：性能优化（否则你只是在更快地产生错误）
```

使用这棵树的正确方式是：

1. 先用真实流量抽样，测出基线（至少有 P50/P95、成本、失败率）。
2. 把你最痛的指标放在最上面（成本/延迟/吞吐/质量）。
3. 选第一刀与第二刀，每刀都必须能在 1-2 周内给出指标结论。

### 2.2.2.1 关键优化技术的生效条件清单

> **使用方法**：在做技术选型时，对照此表判断某项技术是否值得投入。

#### PD 分离（Prefill-Decode Disaggregation）

| 条件 | 判断方法 | 预期收益 | 工程复杂度 |
|------|----------|----------|------------|
| P95 TTFT > 500ms | 监控 TTFT 分布 | TTFT 降 30-50% | 中等 |
| prefill/decode GPU 利用率差异 > 30% | nvidia-smi 观察 | 分离后更稳定 | 中等 |
| 流量中长 prompt (> 2K tokens) 占比 > 30% | 日志分析 | 分离收益明显 | 中等 |
| 混合负载（在线+离线推理） | 业务特征 | **必须分离** | 高 |

> 若以上条件均不满足，PD 分离的工程复杂度可能高于收益。建议先用 vLLM 或 SGLang 的分离模式做 A/B 测试。

#### Prefix Caching

| 条件 | 判断方法 | 预期收益 | 工程复杂度 |
|------|----------|----------|------------|
| 系统提示词长度 > 100 tokens | 统计 system prompt | 首次后 TTFT 降 50%+ | 低 |
| 系统提示词重复率 > 80% | 日志分析 | 显著降低 prefill | 低 |
| RAG 场景（固定知识前缀） | 业务特征 | 每次请求节省前缀计算 | 低 |
| 多轮对话（共享历史） | 业务特征 | 对话内 KV 复用 | 中等 |

> Prefix Caching 几乎无副作用，建议在生产环境默认启用（vLLM 0.6.0+：`--enable-prefix-caching`）。

#### 连续批处理（Continuous Batching）

| 条件 | 判断方法 | 预期收益 | 工程复杂度 |
|------|----------|----------|------------|
| 并发请求数 > 4 | 监控并发 | 吞吐量提升 2-5× | 低 |
| 请求长度差异大 | 统计 prompt/output 长度分布 | 减少 padding 浪费 | 低 |
| 峰值流量波动明显 | 监控 QPS 曲线 | 更好地利用 GPU | 中等 |

> vLLM/SGLang 默认启用连续批处理。这是"稳赚不赔"的优化。

#### 量化（Quantization）

| 条件 | 判断方法 | 预期收益 | 工程复杂度 |
|------|----------|----------|------------|
| 显存不足（OOM 频繁） | 监控 OOM 次数 | 显存需求降 50-75% | 低 |
| 成本敏感 | 业务成本结构 | GPU 需求降 30-50% | 低 |
| 延迟敏感 | P95 延迟过高 | TPOT 改善 20-30% | 低 |
| 质量敏感（复杂推理） | 业务对准确率要求 | **谨慎使用** | - |

> 量化需注意：
> - 复杂推理任务（数学、代码）：建议 FP8 而非 INT8/INT4
> - 上线前必须在业务回归集上验证质量损失 < 1%

#### 投机采样（Speculative Decoding）

| 条件 | 判断方法 | 预期收益 | 工程复杂度 |
|------|----------|----------|------------|
| Decode 阶段占比较高 | 延迟分解 | TPOT 改善 1.5-2× | 高 |
| 长输出场景 (> 512 tokens) | 业务特征 | 收益更明显 | 高 |
| 有小模型可用作 draft | 资源情况 | 成本不变，提速 | 中等 |

> 投机采样工程复杂度高，建议在完成基础优化（批处理、量化、缓存）后再考虑。
