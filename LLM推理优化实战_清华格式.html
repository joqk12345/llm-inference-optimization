LLM推理优化实战

编著

内容简介

本书系统讲解大型语言模型（LLM，Large Language Model）推理优化的核心技术与工程实践，旨在帮助工程师建立系统级的推理优化能力。

核心内容分为四个部分：

第一部分（第1-2章）：建立推理优化的全局视角，理解为什么优化是必要的，以及技术演进的方向。

第二部分（第3-5章）：夯实基础，包括GPU硬件基础、推理环境搭建、以及LLM推理的核心原理（Prefill/Decode、Attention、KV Cache）。

第三部分（第6-9章）：深入核心技术，包括KV Cache优化（PagedAttention、Prefix Caching）、请求调度策略、量化技术（FP8/INT8/INT4）、投机采样等。

第四部分（第10-11章）：生产实践与前沿话题，包括生产环境部署、监控运维、成本优化，以及PD分离、MoE等高级主题。

本书特色：

1. 从第一性原理出发，帮助读者理解“为什么”而非仅是“怎么做”；

2. 强调工程可落地性，提供生产级配置示例与监控告警方案；

3. 量化思维贯穿始终，每个优化都配有预期收益的量化参考。

读者对象：平台工程师、ML Infra工程师、后端工程师、全栈工程师、算法工程师、技术决策者。

前言

编写意图

随着大型语言模型（LLM，Large Language Model）在工业界的广泛应用，推理（Inference）已经从“能跑”变成了“要用得起”。训练一个模型可能只需要几个月，但推理服务要持续运行几年——每一个token的计算、每一次请求的延迟、每一字节的带宽，都在持续消耗资源。

我们编写这本书的初衷，是希望帮助工程师们建立起系统级的推理优化能力——不是学会某个框架的使用技巧，而是理解为什么优化有效，在什么条件下有效、以及如何衡量优化效果。

本书特点

1. 第一性原理出发：不只讲“怎么做”，更讲“为什么这样做”。理解原理是灵活应用的前提。

2. 工程导向：每一个技术点都配有生产环境可用的配置示例、监控指标和故障排查指南。

3. 量化思维：强调用数据说话——优化前后的延迟、吞吐、成本变化，都需要可度量的对比。

4. 场景适配：不同业务场景的优化重点不同，本书帮助读者建立“判断框架”而非机械套用。

适用范围

平台工程师/Infra工程师：需要搭建和维护推理平台，关注成本、稳定性与可扩展性。

后端工程师/全栈工程师：将LLM接入业务，关注API延迟、失败率与用户体验。

ML工程师/算法工程师：关注模型质量与系统约束的平衡。

技术决策者（CTO、架构师）：需要评估技术路线与投资回报。

编写方法

本书采用“问题驱动”的组织方式：每章从实际问题出发，先分析问题根源，再介绍解决方案，最后给出工程实践指南。

各章内容相对独立，读者可以根据自己的需求选择性阅读，也可以按顺序通读形成系统认知。

第1章 重新理解推理这件事

1.1 训练vs推理：就像“研发”和“生产”

想象一下你开一家工厂：

训练就像研发新产品——你一次性投入资金（几百万美元），建实验室、买设备、雇科学家，花几个月做出一个产品原型。产品做出来之后，这个投入就结束了。

推理就像量产销售——产品定型后，每卖出一件都有成本（GPU电费、工程师工资、运维费用），而且你卖得越多，成本就越高。

这就是训练和推理的本质区别：

维度训练推理
投入模式一次性（几个月）持续性（每天）
成本规模数百万美元一次每月数十万-数百万美元
决策频率几个月做一次每天都在做
优化空间相对固定可持续优化

为什么这很重要

很多团队在推理上的投入远超训练——不是因为他们笨，而是因为这就是现实。训练模型是“少数公司做、少数几次”的事，但推理是“所有公司都在做、每天都在做”的事。

我们自己的经历：2023年我们团队做过一个项目，模型训练花了50万美元，但上线第一年的推理成本花了200万——这是大多数团队没有意识到的“冰山下的成本”。

1.2 技术可行：优化空间与成本结构

1.2.1 仍有下降空间

推理系统里还有大量“结构性浪费”：

1) 精度冗余：FP8/INT4足够却用FP16

2) 重复计算：系统提示词重复计算

3) 资源错配：Prefill/Decode混用同一GPU

4) 调度低效：batch组织不合理

1.2.2 成本结构拆解

单位请求成本 = (GPU成本 + 显存占用 + 网络成本 + 运维摊销) / 有效请求数

1.3 案例启示：从原理到实践

1.3.1 客服/知识问答场景

目标：降低单次问答成本，提高并发与响应速度

常见抓手：批量调度、KV Cache、量化、负载均衡

可衡量指标：单位成本、P95延迟、并发上限、拒绝率

核心要点：

1. 推理优化是AI从“能力展示”走向“规模生产”的分水岭

2. 推理成本结构：GPU + 显存 + 网络 + 运维，可拆解可优化

第2章 技术全景与趋势

2.1 技术趋势概览

2.1.1 DeepSeek V3：MoE范式的代表性案例

MoE（Mixture of Experts，混合专家）的核心承诺是：总参数很大，但每次只激活一小部分参数，从而把“能力上限”和“计算成本”部分解耦。

2.1.2 PD分离（Prefill-Decode分离）

PD分离（Prefill/Decode Disaggregation）之所以成为趋势，是因为它对应了推理的一个结构性事实：prefill和decode是两种完全不同的工作负载。

prefill更像计算密集型：大的矩阵乘、Attention，适合吞吐型batch。

decode更像带宽/延迟敏感型：每步都要访问KV Cache，且请求之间彼此干扰大。

2.1.3 RL Info的兴起

RL（RLHF/RLAIF/各种后训练）的核心资源消耗之一是大规模rollout（采样/生成），本质上就是“以训练目的组织的推理”。

2.1.4 Agent与多模态的爆发

当系统从“单轮问答”变成“多步任务”（检索、工具调用、计划与执行），推理系统的瓶颈会发生明显迁移：上下文与工作流会比模型本身更先决定体验和成本。

2.2 五大优化方向速览

方向1：系统架构优化

关注点：整体架构设计、硬件选型、工作流拆分、隔离与弹性。

典型技术：PD分离、网关与路由、服务拆分、混部与容灾、多模型部署。

适用场景：一切生产系统，尤其是P95/P99与稳定性优先的系统。

方向2：推理引擎优化

关注点：框架选择与配置、内核效率、算子融合与内存管理。

典型技术：vLLM/SGLang/TensorRT-LLM，Attention内核优化、paged/chunked KV。

方向3：模型优化

关注点：压缩、结构、路由与能力-成本权衡。

典型技术：量化（FP16→INT8/INT4/FP8）、蒸馏、剪枝、MoE、分级模型路由。

方向4：请求调度优化

关注点：如何组织请求与token的计算顺序，提升GPU有效利用率。

典型技术：连续批处理、优先级队列、chunking、不同流量形态的隔离。

方向5：缓存与复用

关注点：避免重复计算，让相似请求共享计算结果。

典型技术：前缀缓存、会话缓存、模板化上下文复用、跨请求token复用。

2.3 谁应该读这本书

1) 平台工程师/Infra工程师：负责搭建与维护推理平台，关注成本、稳定性与可扩展性。

2) 后端工程师/全栈工程师：把LLM接到业务里，关注API设计、延迟、失败率与体验。

3) ML工程师/算法工程师：更关心模型质量，但会越来越频繁地被问到：能否更便宜、更快、更稳定？

4) 技术决策者（CTO、架构师）：要对成本、风险与路线负责，不需要写CUDA，但必须能做正确决策。

第3章 GPU基础

3.1 CPU vs GPU：本质差异

类比：数学教授vs小学生团队

CPU像一位数学教授——单次计算能力极强，但只能一个个解决问题。

GPU像一支小学生团队——每个人（CUDA core）只能做简单算术，但团队合作（并行）能快速完成大量简单任务。

为什么GPU适合矩阵运算

Transformer的核心运算都是矩阵乘法：

Y = X × W

这正好符合GPU的并行特性——大量相同的运算可以同时执行。

GPU不适合的任务类型

1) 复杂分支逻辑（如频繁的条件判断）

2) 小数据量的随机访问

3) 高度依赖单线程性能的任务

3.2 GPU架构详解

3.2.1 流式多处理器（SM）：GPU的核心单元

SM（Streaming Multiprocessor）是GPU的计算核心，每个SM包含：

  • CUDA Cores（FP32/INT32）：通用计算单元
  • Tensor Cores：矩阵运算专用加速器
  • Register File：寄存器
  • Shared Memory / L1 Cache：高速共享内存
  • Warp Scheduler：线程调度器
  • SFU：特殊函数单元
单元类型功能精度吞吐量适用场景
CUDA Cores通用计算FP32/INT321× base非矩阵运算
Tensor Cores矩阵乘法FP16/BF16/FP8/FP416-64× baseGEMM、Attention

3.2.2 显存（VRAM）：容量vs带宽

GPUHBM版本容量带宽
A100HBM2e80 GB2.0 TB/s
H100HBM380 GB3.35 TB/s
H200HBM3e141 GB4.8 TB/s
B200HBM3e192 GB8 TB/s

带宽为什么重要

LLM推理 = Memory-bound operation

每1个token生成：
- 读取模型权重: 80 GB (Llama-3-70B @ FP16)
- 读取KV Cache: 数MB到数GB
- 写入新token: ~1 KB

如果带宽 = 3.35 TB/s (H100):
- 读取80 GB需要: 80 GB / 3.35 TB/s = 23.9 ms
- 这是理论下限！(实际会更慢)

3.3 显存计算公式

模型权重计算

模型显存 ≈ 参数数量 × 每参数字节数

例如Llama-2-7B:
- 参数数量: 7B (70亿)
- FP16: 2 bytes/参数
- 总显存 ≈ 7B × 2 = 14 GB

KV Cache显存占用

KV Cache ≈ 2 × num_layers × num_heads × head_dim × seq_len × bytes

例如Llama-2-7B (32层, 32头, 128维, 4096序列长度):
= 2 × 32 × 32 × 128 × 4096 × 2 bytes
≈ 2 GB

第5章 LLM推理基础

5.1 训练vs推理：工作负载的本质差异

训练：计算密集型的并行工作负载

训练流程：

输入数据 → Forward Pass → Loss Calculation → Backward Pass → Weight Update

特点：

  • Compute-bound：计算密集型
  • 高度并行：整个batch同时处理
  • GPU利用率高：矩阵运算充分利用GPU
  • 极其昂贵：需要大规模GPU集群

推理：内存带宽密集型的串行工作负载

推理流程：

用户Prompt → Prefill (并行) → Decode (串行) → 返回结果

特点：

  • Memory-bound：内存带宽密集型
  • 串行生成：Decode阶段必须逐个生成
  • GPU利用率低：大量时间在移动数据而非计算
  • 持续运行：7×24小时服务

5.2 LLM如何生成文本

5.2.1 自回归生成的基本过程

LLM的本质：fancy next token predictors（花哨的下一个词预测器）

生成过程：

步骤1: 输入整个prompt
输入: "Hello, my name is"
模型处理整个prompt
生成第1个token: " John"

步骤2: 添加新token，再次预测
输入: "Hello, my name is John"
模型处理整个序列
生成第2个token: " and"

步骤3: 重复...
输入: "Hello, my name is John and"
生成第3个token: " I"
...

5.2.2 Prefill阶段：并行处理prompt

定义：处理初始prompt，生成第一个token的阶段

特点：

  • 计算密集型：大量矩阵乘法
  • 可以并行处理：所有token同时计算
  • 时间：TTFT (Time To First Token)，首字延迟

5.2.3 Decode阶段：逐token生成

定义：逐个生成后续token的阶段

特点：

  • 内存带宽密集型：每次只生成1个token
  • 无法并行：必须逐个生成（因果关系）
  • 时间：TBT (Time Between Tokens)，字间延迟

5.3 Attention机制详解

5.3.1 Token的表示：向量与hidden dimension

Tokenization：文本→token序列

文本: "Hello, world!"
↓ Tokenizer
Tokens: [15496, 11, 2159, 0]
       │      │   │    │
     Hello    ,  world  <eos>

Embedding：每个token→d维向量

Tokens: [15496, 11, 2159, 0]
       │      │    │    │
       ▼      ▼    ▼    ▼
Embeddings:
  Token 15496 → [0.12, -0.34, 0.56, ..., 0.78]  (d=4096)
  Token 11    → [-0.23, 0.45, -0.67, ..., 0.89]
  Token 2159 → [0.34, -0.56, 0.78, ..., -0.12]

5.3.2 Query、Key、Value投影

三个权重矩阵：Wq、Wk、Wv

每个token的表示: x

x = embeddings[i]  # shape: [hidden_dim]

# 投影到Q、K、V
Q = x @ Wq  # Query: 这个token想找什么?
K = x @ Wk  # Key: 这个token能提供什么?
V = x @ Wv  # Value: 这个token的实际内容

# Wq, Wk, Wv的形状: [hidden_dim, head_dim]
# Q, K, V的形状: [head_dim]

5.3.3 Attention计算：QK^T与二次复杂度

# 输入: Q, K, V
# 形状: [batch, num_heads, seq_len, head_dim]

# 步骤1: 计算Q @ K^T
scores = Q @ K.transpose(-2, -1)  # [batch, num_heads, seq_len, seq_len]

# 步骤2: 缩放
scores = scores / (head_dim ** 0.5)

# 步骤3: Softmax归一化
attn_weights = torch.softmax(scores, dim=-1)

# 步骤4: 加权求和
output = attn_weights @ V  # [batch, num_heads, seq_len, head_dim]

# 总复杂度: O(n²·d)

关键洞察：Attention的二次复杂度是性能瓶颈！

序列长度n = 1000:
- Attention计算: 1000² × 4096 = 4,096,000,000次运算

序列长度n = 10000:
- Attention计算: 10000² × 4096 = 409,600,000,000次运算 (100倍!)

第6章 KV Cache优化

6.1 Transformer回顾

6.1.1 Attention机制原理

Attention的本质：让每个token能够"看到"并"聚合"其他token的信息。

def attention(Query, Key, Value):
    # 1. 计算相似度
    scores = Query @ Key.T  # [seq_len, seq_len]

    # 2. 归一化
    attn_weights = softmax(scores / sqrt(d_k))

    # 3. 加权求和
    output = attn_weights @ Value  # [seq_len, d_v]

    return output

6.2 KV Cache原理

6.2.1 生成过程的重复计算问题

朴素方法的问题：

步骤1: "The capital of France is"
→ 计算Token 0-6的K、V
→ 生成"Paris"

步骤2: "The capital of France is Paris"
→ 重新计算Token 0-7的K、V ✗ (重复计算!)
→ Token 0-6的K、V又浪费了!

步骤3: "The capital of France is Paris and"
→ 重新计算Token 0-8的K、V ✗✗ (更多重复)
→ Token 0-7的K、V又浪费了!

6.2.2 KV Cache的核心思想

核心洞察：旧token的K、V已经计算过，缓存起来！

Prefill阶段:
输入: "Hello, my name is"
计算:
  Token 0: K0, V0
  Token 1: K1, V1
  Token 2: K2, V2
  Token 3: K3, V3
存储: 缓存所有K, V

Decode阶段 - 第1步:
输入: 新Token 4
计算:
  Token 4: K4, V4
复用: K0, K1, K2, K3 (从缓存读取)
组合: [K0, K1, K2, K3, K4]
      [V0, V1, V2, V3, V4]

6.3 KV Cache实现

6.3.1 PagedAttention原理

灵感来源：操作系统虚拟内存

OS Virtual Memory:  Pages (4KB) + Page Table
vLLM KV Cache:      Blocks (16 tokens) + Block Table

核心概念：

  • Logical blocks：逻辑上的连续序列（用户视角）
  • Physical blocks：GPU内存中的实际块（系统视角）
  • Block table：映射关系（logical → physical）

6.3.2 Block Allocation策略

# vLLM的启动时分配
def allocate_at_startup():
    # 计算可用GPU内存
    gpu_memory = get_gpu_memory()
    # 预分配90%给KV Cache (保留10%给模型weights)
    num_blocks = (gpu_memory * 0.9) / BLOCK_SIZE
    # 创建block pool
    block_pool = BlockPool(num_blocks)
    return block_pool

第7章 请求调度策略

7.1 调度的必要性

为什么需要调度

在线推理天然是“多租户+波动流量”。短请求、长请求、长上下文、峰值并发会同时存在。你不显式做调度，系统就会用排队与阻塞替你“隐式调度”，表现为P95/P99抖动、超时与成本上升。

调度器的目标

目标说明
最小化延迟P50、P95、P99延迟尽可能低
最大化吞吐量在给定硬件上服务更多用户
公平性避免长请求饿死短请求
资源利用GPU利用率尽可能高

7.2 基础调度策略

7.2.1 FIFO (First In First Out)

class FIFOScheduler:
    def __init__(self):
        self.queue = []

    def add_request(self, request):
        self.queue.append(request)

    def schedule(self):
        if self.queue:
            return [self.queue.pop(0)]  # 返回第一个请求
        return []

优点：实现简单、公平、无饥饿

缺点：吞吐量低、GPU利用率偏低、长请求阻塞后续所有请求

7.2.2 静态批处理 (Static Batching)

class StaticBatchScheduler:
    def __init__(self, batch_size=8):
        self.batch_size = batch_size
        self.queue = []

    def add_request(self, request):
        self.queue.append(request)

    def schedule(self):
        if len(self.queue) >= self.batch_size:
            batch = self.queue[:self.batch_size]
            self.queue = self.queue[self.batch_size:]
            return batch
        return []

7.3 Continuous Batching原理

核心思想：动态调整batch，在请求完成时立即释放资源，允许新请求加入。

# vLLM Continuous Batching示例

# 传统批处理：等所有请求完成 → 释放 → 加入新请求
# 连续批处理：任一请求完成 → 立即加入新请求

# 伪代码
while True:
    # 1. 执行当前batch
    results = execute_batch(current_batch)

    # 2. 收集已完成的请求
    completed = [r for r in current_batch if r.is_done()]

    # 3. 释放完成请求的资源
    for r in completed:
        free_resources(r)

    # 4. 立即加入等待队列中的新请求
    waiting = get_waiting_requests()
    current_batch = add_to_batch(completed, waiting)

第8章 量化技术

8.1 量化基础

为什么需要量化

LLM的模型权重和激活值通常使用FP32（32位浮点）或FP16（16位浮点）存储和计算。但很多场景下：

  • 不需要这么高的精度
  • 更低的精度可以大幅减少显存和计算
  • 硬件对低精度有专门加速

量化的核心：用更少的位表示数值，同时尽量减少信息损失。

8.2 量化方法分类

8.2.1 训练后量化 (PTQ)

训练后量化（Post-Training Quantization，PTQ）是在模型训练完成后直接进行量化。

优点：无需重新训练，快速部署

缺点：精度可能有损失

8.2.2 量化感知训练 (QAT)

量化感知训练（Quantization-Aware Training，QAT）在训练过程中模拟量化效果。

优点：精度更好

缺点：需要重新训练，时间成本高

8.3 常用量化格式

8.3.1 FP16 vs BF16

格式范围精度稳定性推荐度
FP16小高一般（可能下溢）⭐⭐⭐
BF16大中好（不易下溢）⭐⭐⭐⭐⭐

8.3.2 FP8 / NVFP4

FP8 (8位浮点) 有两种格式：

E4M3 (4 bit指数, 3 bit尾数):
  用于训练
  范围: ±448
  精度: 类似FP16

E5M2 (5 bit指数, 2 bit尾数):
  用于推理
  范围: ±57344
  精度: 低于E4M3

8.4 流行的量化框架

AWQ (Activation-aware Quantization)

原理：基于激活值的重要性来量化权重，重要的权重保持高精度。

步骤:
1. 收集激活值统计
2. 计算权重重要性
3. 非均匀量化（重要权重精度高）

GPTQ (Gradient-based Post-Training Quantization)

原理：基于梯度的后训练量化，使用二阶信息来最小化量化误差。

8.5 KV Cache量化

INT8 KV Cache

FP16 → INT8:
  FP16: 每个元素2 bytes
  INT8: 每个元素1 byte

  KV Cache大小减半!

第9章 投机采样

9.1 什么是投机采样

投机采样（Speculative Decoding）是一种“猜测-验证”策略：使用一个小模型快速生成多个候选token，再用大模型验证和修正。

9.2 投机采样的原理

传统自回归:
大模型 → token1 → token2 → token3 → ... (逐个生成)

投机采样:
小模型(draft) → [t1, t2, t3, t4, t5]
大模型(verify) → 验证并修正 → 最终输出

核心思想：用小模型的算力换大模型的时间。
